Fly.io is a global application platform running apps on Firecracker micro-VMs (Fly Machines) across 30+ regions with anycast routing. It excels at latency-sensitive workloads, AI agent sandboxes (Sprites), and apps that need to run close to users globally.

## Provider Overview

### Fly.io Characteristics

- **Deployment type**: Managed PaaS on Firecracker micro-VMs (Fly Machines)
- **Networking**: Global anycast — traffic routed to nearest healthy region automatically
- **Regions**: 30+ worldwide (ams, cdg, dfw, fra, hkg, iad, lax, lhr, nrt, ord, sea, sin, syd, yyz, etc.)
- **Languages**: Any Docker-compatible app — Node.js, Python, Go, Ruby, Elixir, Rust, PHP, Java
- **Databases**: Managed Postgres (Fly Postgres), Redis (Upstash), SQLite (LiteFS for distributed)
- **AI sandboxes**: Sprites — isolated Fly Machines for running untrusted AI agent code
- **Auto-stop**: Machines stop when idle, start on first request (pay only for active time)
- **Volumes**: Persistent NVMe storage attached to Machines

### Best Use Cases

- **Global low-latency apps** — anycast puts compute near users automatically
- **AI agent sandboxes** — Sprites provide isolated execution environments
- **Elixir/Phoenix** — first-class support, excellent for distributed systems
- **Full-stack apps** — deploy frontend + backend + database in one platform
- **Cost-sensitive workloads** — auto-stop/start eliminates idle compute costs
- **Multi-region databases** — LiteFS for distributed SQLite, Fly Postgres with replicas

### When NOT to Use Fly.io

- **Serverless functions** — use Cloudflare Workers or Vercel Edge Functions instead
- **Static sites only** — Cloudflare Pages or Vercel are simpler and cheaper
- **Kubernetes-native workloads** — use GKE/EKS/AKS directly
- **Windows containers** — Fly.io is Linux-only

## fly.toml Configuration

The `fly.toml` file defines your app's configuration. It lives in the repo root.

```toml
# fly.toml — generated by `fly launch`, customised manually

app = "my-app-name"
primary_region = "lhr"  # London — change to nearest region

[build]
  # Option 1: Dockerfile (recommended)
  dockerfile = "Dockerfile"

  # Option 2: Buildpack (auto-detect language)
  # builder = "heroku/buildpacks:20"

[env]
  PORT = "8080"
  NODE_ENV = "production"

[http_service]
  internal_port = 8080
  force_https = true
  auto_stop_machines = true   # Stop when idle (cost saving)
  auto_start_machines = true  # Start on first request
  min_machines_running = 0    # 0 = full auto-stop (cheapest)
  # min_machines_running = 1  # 1 = always-on (no cold starts)

  [http_service.concurrency]
    type = "requests"
    hard_limit = 250
    soft_limit = 200

[[vm]]
  memory = "256mb"
  cpu_kind = "shared"
  cpus = 1

# Persistent volume mount
[[mounts]]
  source = "myapp_data"
  destination = "/data"

# Health check
[[http_service.checks]]
  grace_period = "10s"
  interval = "30s"
  method = "GET"
  path = "/health"
  protocol = "http"
  timeout = "5s"
```

### Key fly.toml Options

| Option | Values | Notes |
|--------|--------|-------|
| `primary_region` | `lhr`, `iad`, `nrt`, etc. | Where volumes and primary Machine live |
| `auto_stop_machines` | `true`/`false` | Stop idle Machines (saves cost) |
| `min_machines_running` | `0`–N | 0 = full auto-stop, 1+ = always-on |
| `cpu_kind` | `shared`, `performance` | Shared = burstable, Performance = dedicated |
| `memory` | `256mb`–`64gb` | RAM per Machine |

## Fly Machines

Fly Machines are Firecracker micro-VMs — lightweight, fast-starting (sub-second), and billed per second of active time.

### Machine Lifecycle

```bash
# List machines for an app
fly machines list --app my-app

# Start a stopped machine
fly machines start <machine-id> --app my-app

# Stop a running machine
fly machines stop <machine-id> --app my-app

# Destroy a machine permanently
fly machines destroy <machine-id> --app my-app --force

# SSH into a machine
fly ssh console --app my-app

# Run a one-off command
fly ssh console --app my-app --command "rails db:migrate"
```

### Machine Sizes

| Size | CPU | RAM | Use case |
|------|-----|-----|----------|
| `shared-cpu-1x` | 1 shared | 256 MB | Dev, low-traffic |
| `shared-cpu-2x` | 2 shared | 512 MB | Light production |
| `shared-cpu-4x` | 4 shared | 1 GB | Medium workloads |
| `performance-1x` | 1 dedicated | 2 GB | CPU-intensive |
| `performance-2x` | 2 dedicated | 4 GB | High-throughput |
| `performance-4x` | 4 dedicated | 8 GB | Heavy compute |
| `performance-8x` | 8 dedicated | 16 GB | AI inference |

## Deployment

### Initial Setup

```bash
# Launch a new app (interactive — creates fly.toml)
fly launch

# Launch with specific options (non-interactive)
fly launch --name my-app --region lhr --no-deploy

# Deploy after launch
fly deploy
```

### Deploy Workflow

```bash
# Deploy current directory (uses fly.toml)
fly deploy

# Deploy with specific image
fly deploy --image registry.fly.io/my-app:latest

# Deploy to specific region only
fly deploy --region lhr

# Deploy with build args
fly deploy --build-arg NODE_VERSION=20

# Watch deployment logs
fly logs --app my-app

# Check deployment status
fly status --app my-app
```

### Rollback

```bash
# List releases
fly releases --app my-app

# Rollback to previous release
fly deploy --image <previous-image-ref>
```

## Scaling

```bash
# Scale machine count (horizontal)
fly scale count 3 --app my-app

# Scale machine count per region
fly scale count 2 --region lhr --app my-app
fly scale count 1 --region iad --app my-app

# Scale machine size (vertical)
fly scale vm performance-2x --app my-app

# Scale memory
fly scale memory 1024 --app my-app

# Show current scale
fly scale show --app my-app
```

## Secrets Management

Secrets are injected as environment variables into Machines. They are encrypted at rest and never appear in logs.

```bash
# Set a secret (value read from stdin — never pass as argument)
echo "my-secret-value" | fly secrets set MY_SECRET=- --app my-app

# Set multiple secrets from a file
fly secrets import --app my-app < .env.production

# List secret names (values never shown)
fly secrets list --app my-app

# Remove a secret
fly secrets unset MY_SECRET --app my-app
```

**Security note**: Never pass secret values as command arguments — they appear in shell history and process lists. Use stdin (`=- `) or `fly secrets import` from a file.

## Volumes (Persistent Storage)

Fly Volumes are NVMe SSDs attached to Machines. They are region-specific — a volume in `lhr` can only attach to a Machine in `lhr`.

```bash
# Create a volume
fly volumes create myapp_data --size 10 --region lhr --app my-app

# List volumes
fly volumes list --app my-app

# Extend a volume (increase size only — cannot shrink)
fly volumes extend <volume-id> --size 20 --app my-app

# Show volume details
fly volumes show <volume-id> --app my-app

# Destroy a volume (IRREVERSIBLE — data lost)
fly volumes destroy <volume-id> --app my-app
```

### Volume Best Practices

- Always create volumes before deploying apps that need them
- Use `[[mounts]]` in `fly.toml` to attach volumes to Machines
- For multi-region apps with shared data, use Fly Postgres or LiteFS instead of volumes
- Snapshot volumes before extending or migrating

## Databases

### Fly Postgres

Fly Postgres is a managed PostgreSQL cluster running on Fly Machines (not a fully managed service — you manage upgrades).

```bash
# Create a Postgres cluster
fly postgres create --name my-app-db --region lhr

# Attach to an app (sets DATABASE_URL secret automatically)
fly postgres attach my-app-db --app my-app

# Connect to Postgres (via proxy)
fly postgres connect --app my-app-db

# List databases
fly postgres db list --app my-app-db

# Check cluster status
fly status --app my-app-db
```

### Upstash Redis

Upstash provides serverless Redis on Fly.io — pay per request, no idle cost.

```bash
# Create Redis instance
fly redis create --name my-app-redis --region lhr

# Attach to app (sets REDIS_URL secret)
fly redis attach my-app-redis --app my-app

# List Redis instances
fly redis list
```

## Multi-Region Deployment

```bash
fly regions add iad --app my-app

# Remove a region
fly regions remove iad --app my-app

# List active regions
fly regions list --app my-app

# Set backup regions (failover)
fly regions backup iad --app my-app
```

### Multi-Region Database Pattern

For apps needing low-latency reads globally:

```toml
# fly.toml — multi-region with Postgres read replicas
[env]
  PRIMARY_REGION = "lhr"

# Fly Postgres handles read replica routing automatically
# when DATABASE_URL points to the cluster
```

## Sprites (AI Agent Sandboxes)

Sprites are isolated Fly Machines designed for running untrusted AI agent code — each agent gets its own ephemeral VM.

```bash
# Sprites are created via the Fly Machines API
# Use the fly-io-helper.sh machines subcommand or the REST API directly

# Create a Machine for an AI agent (ephemeral)
fly machines run my-sandbox-image \
  --app my-sprites-app \
  --region lhr \
  --vm-size shared-cpu-1x \
  --env AGENT_ID=agent-123 \
  --restart no  # Don't restart — ephemeral

# Destroy after use
fly machines destroy <machine-id> --app my-sprites-app --force
```

### Free Allowances (Hobby plan)

- 3 shared-cpu-1x VMs with 256 MB RAM
- 3 GB persistent volume storage
- 160 GB outbound data transfer
- Shared IPv4 address (dedicated IPv4: $2/month)

### Cost Optimisation

```toml
# fly.toml — cost-optimised config
[http_service]
  auto_stop_machines = true
  auto_start_machines = true
  min_machines_running = 0  # Full auto-stop — pay only when active

[[vm]]
  memory = "256mb"
  cpu_kind = "shared"
  cpus = 1
```

**Auto-stop savings**: An app with 8 hours/day active time on `shared-cpu-1x` costs ~$0.65/month vs $1.94/month always-on — 67% savings.

### Pricing vs Competitors

| Platform | Shared CPU 1x / 256 MB | Always-on | Auto-stop | Global anycast |
|----------|------------------------|-----------|-----------|----------------|
| Fly.io | ~$1.94/mo | Yes | Yes | Yes |

Fly.io is the most cost-effective option for always-on global apps with auto-stop. Vercel/Cloudflare Workers are cheaper for pure serverless/edge functions.

## Common Workflows

### Deploy a Node.js App

```bash
# 1. Create Dockerfile if not present
cat > Dockerfile << 'EOF'
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 8080
CMD ["node", "server.js"]
EOF

# 2. Launch app
fly launch --name my-node-app --region lhr

# 3. Set secrets
echo "my-db-password" | fly secrets set DB_PASSWORD=- --app my-node-app

# 4. Deploy
fly deploy --app my-node-app

# 5. Open in browser
fly open --app my-node-app
```

### Deploy with Persistent Storage

```bash
# 1. Create volume first
fly volumes create app_data --size 5 --region lhr --app my-app

# 2. Add mount to fly.toml
# [[mounts]]
#   source = "app_data"
#   destination = "/data"

# 3. Deploy
fly deploy --app my-app
```

### Zero-Downtime Deploy

Fly.io performs rolling deploys by default — new Machines start before old ones stop.

```bash
# Deploy with explicit strategy
fly deploy --strategy rolling --app my-app

# Or canary (deploy to 1 machine first)
fly deploy --strategy canary --app my-app
```
